# Rename concatenation operator from `..` to `++`

## Objective

Replace the `..` concatenation operator with `++` throughout the entire codebase: tokenizer, parser, compiler, tests, examples, documentation, and the pending arrays plan. When done, `"a" ++ "b"` produces `"ab"`, and `..` is no longer recognized as an operator (freeing it for future range/slice syntax).

## Acceptance criteria

- [ ] `++` tokenizes as a single `TOK_OPERATOR` with value `"++"`, even without spaces (e.g. `"a"++"b"`)
- [ ] `+` still tokenizes as a solo symbol for arithmetic (e.g. `a+b` still works)
- [ ] `++` has precedence 5 and is right-associative (identical to today's `..`)
- [ ] `"hello" ++ " world"` evaluates to `"hello world"`
- [ ] Auto-coercion still works: `"score: " ++ 42` evaluates to `"score: 42"`
- [ ] Chaining still works: `"a" ++ "b" ++ "c"` evaluates to `"abc"`
- [ ] `..` is no longer a recognized infix operator (returns precedence 0, parsed as an error)
- [ ] All existing tests pass with updated syntax
- [ ] `make test && make check` pass with zero failures and zero warnings

## Dependencies

None. This is a pure rename with no dependency on other tasks in `plans/doing/`.

## Constraints and non-goals

- **No behavioral changes.** Precedence, associativity, auto-coercion, and the `OP_CONCAT` bytecode opcode all stay the same. Only the surface syntax changes.
- **Do not rename `OP_CONCAT`** in `chunk.h` or elsewhere. The opcode name is an internal detail.
- **Do not add new features.** This task is strictly a rename.
- **`.expected` files** must be regenerated by running each changed example through the interpreter, per AGENTS.md rules.
- Completed plan files in `plans/done/` are historical records — do not modify them.

## Steps

### Step 1 — Update tokenizer to recognize `++`

In `src/tokenizer.c`, function `read_operator()`:

- `+` is currently listed as a solo symbol in `is_solo_symbol()`. Add a special case: before emitting `+` as a solo symbol, peek at the next character. If it is also `+`, consume both characters and emit a single `TOK_OPERATOR` with value `"++"`.
- Update the two comments in `read_number()` that mention `'..' concat operator` to say `'++' concat operator` (or just `concat operator`).

### Step 2 — Update tokenizer tests

In `tests/test_tokenizer.c`:

- Rename the `String concatenation operator (..)` test group heading to reference `++`.
- Update `test_concat_op_basic`: change input from `".."` to `"++"`, expect `TOK_OPERATOR` with value `"++"`.
- Update `test_concat_op_in_expr`: change input from `"a" .. "b"` to `"a" ++ "b"`, expect the `++` operator token.
- Update `test_concat_op_no_spaces`: change input from `"a".."b"` to `"a"++"b"`, expect the `++` operator token.
- Update `test_concat_integer_before_dot_dot`: this tests that `5..10` tokenizes as `5`, `..`, `10`. Change to test `5++10` → `5`, `++`, `10`. (The original test's purpose was ensuring the number parser doesn't swallow `..`; now that concatenation is `++`, the number parser has no conflict with `+` anyway, but the test still verifies that `++` tokenizes correctly after a number.)
- Remove or update any test that specifically expects `..` to tokenize as `TOK_OPERATOR`. If `..` should no longer be a valid operator token, add a test verifying that `..` now tokenizes as two separate `.` operator tokens (since `.` is a symbol char and not a solo symbol, `..` will naturally tokenize as a single two-char operator token of value `".."`; this is fine — the parser will simply not recognize it as an infix operator).
- Register any new/renamed tests in the test runner function.

### Step 3 — Update parser

- `is_right_assoc()`: change the condition `op[0] == '.' && op[1] == '.'` to `op[0] == '+' && op[1] == '+'`.
- Update the comment on `is_right_assoc()` that says `.. (concatenation)` to say `++ (concatenation)`.
