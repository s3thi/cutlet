# `in` Membership Operator

## Objective

Add `in` as a binary keyword operator for membership testing across maps, arrays, and strings. Add `not in` as syntactic sugar that desugars to the same AST as `not (x in y)`.

When done:

- `"name" in {name: "alice"}` → true (map key membership).
- `42 in [1, 2, 42]` → true (array element membership).
- `"lo" in "hello"` → true (substring search).
- `10 not in [1, 2, 3]` → true (`not in` sugar).
- `not 10 in [1, 2, 3]` → true (equivalent — `not` has lower precedence than `in`).
- `in` is non-associative (like other comparison operators): `1 in [1] in [[1]]` is a parse error.
- `make test && make check` pass.

## Acceptance criteria

- [ ] `"in"` is a reserved keyword.
- [ ] `"in"` is recognized as a keyword-based infix operator at comparison precedence (4).
- [ ] `"in"` is non-associative (chaining is a parse error).
- [ ] `x not in y` desugars to `not (x in y)` — same AST as the explicit form.
- [ ] Compiler emits `OP_IN` for `AST_BINOP "in"`.
- [ ] VM `OP_IN` handler supports `VAL_MAP` (key existence), `VAL_ARRAY` (element search), `VAL_STRING` (substring search).
- [ ] Parser tests, compiler tests, and VM integration tests all pass.
- [ ] `make test && make check` pass.

## Dependencies

- **Arrays plan** (`plans/doing/20260219150840-arrays.md`): must be complete. Provides `VAL_ARRAY` and `ObjArray` for array membership testing.
- **Maps plan** (`plans/doing/20260223173805-maps.md`): must be complete. Provides `VAL_MAP`, `ObjMap`, `obj_map_has()`, and `value_equal()`.

## Constraints and non-goals

- **No `contains()` builtin.** `in` replaces the need for it.
- **`has_key()` remains.** It's still useful in higher-order contexts (passing to functions), but `in` is the idiomatic way to check membership interactively.
- **No `not in` as a distinct operator.** It desugars to `[UNARY not [BINOP in ...]]` in the parser. The compiler and VM never see "not in" — they see `OP_NOT` applied to `OP_IN`.

## Design

### `in` operator

`in` is a binary keyword operator at comparison precedence (level 4, same as `==`, `<`, `>`, etc.). It is non-associative — chaining like `x in y in z` is a parse error, same as `1 < 2 < 3`.

```cutlet
# Map: checks key existence
"name" in {name: "alice", age: 30}     # => true
"email" in {name: "alice"}              # => false
1 in {[1]: "one", [2]: "two"}          # => true  (non-string keys)
true in {[true]: "yes"}                 # => true

# Array: checks element membership (linear scan, equality-based)
42 in [1, 2, 42]                        # => true
"x" in ["a", "b", "c"]                 # => false
true in [1, true, "hi"]                 # => true

# String: checks substring existence
"lo" in "hello"                         # => true
"xyz" in "hello"                        # => false
"" in "hello"                           # => true  (empty string is in everything)
"hello" in "hello"                      # => true  (string is in itself)
```

Type rules for the right operand:
- `VAL_MAP`: left operand is the key to search for. Uses `obj_map_has()` / `value_equal()`.
- `VAL_ARRAY`: left operand is the value to search for. Linear scan using `value_equal()`.
- `VAL_STRING`: left operand must also be `VAL_STRING`. Uses C `strstr()` for substring search. If left is not a string → runtime error.
- Other types on the right: runtime error "cannot use 'in' with \<type\>".

### `not in` sugar

```cutlet
10 not in [1, 2, 3]                    # => true
"z" not in {a: 1, b: 2}               # => true
"xyz" not in "hello"                   # => true
```

The parser recognizes `expr not in expr` and desugars it to `not (expr in expr)`. The AST is identical to writing `not expr in expr` explicitly:

```
10 not in [1, 2, 3]
→ AST: [UNARY not [BINOP in [NUMBER 10] [ARRAY ...]]]

not 10 in [1, 2, 3]
→ AST: [UNARY not [BINOP in [NUMBER 10] [ARRAY ...]]]
```

Same AST, same bytecode, same result. The `not in` form is purely a parser convenience.

### Precedence and associativity

`in` sits at comparison precedence (4), so it binds tighter than `and`/`or`/`not` but looser than arithmetic:

```cutlet
x + 1 in [1, 2, 3]         # => (x + 1) in [1, 2, 3]
"a" in m and "b" in m       # => ("a" in m) and ("b" in m)
not "z" in m                # => not ("z" in m)
```

Non-associative: `1 in [1] in [[1]]` is a parse error, same as `1 < 2 < 3`.

## Architecture changes

### New opcode

| Opcode | Operand | Stack effect | Description |
|--------|---------|-------------|-------------|
| `OP_IN` | — | pop 2 (needle, haystack), push 1 bool | Membership test |

### Parser changes

1. Add `"in"` to `is_reserved_keyword()`.
2. Add `"in"` to `is_keyword_infix()` (alongside `"and"` and `"or"`).
3. Add `"in"` to `infix_precedence()` returning 4 (comparison level).
4. Add `"in"` to `is_comparison_op()` (for non-associativity enforcement).
5. Add `not in` handling in `parse_expr()`: when the current token is `not` and the next token is `in`, consume both, parse the right operand at `in` precedence, construct `[BINOP in left right]`, then wrap in `[UNARY not ...]`.

For point 5, the logic in `parse_expr()`:
- Currently the while loop checks `p->current.type == TOK_OPERATOR || is_keyword_infix(&p->current)`.
- Add a third condition: `token_is_keyword(&p->current, "not")` — but only when the next token is `in`. Peek ahead to confirm before entering the `not in` path.
- Inside the `not in` path: advance past `not`, advance past `in`, parse right with `in`'s precedence (5, i.e. prec + 1 for left-associativity — though non-associativity means it won't chain anyway), build `AST_BINOP "in"`, wrap in `AST_UNARY "not"`.

### Compiler changes

In `compile_binop()` (or wherever `AST_BINOP` is compiled): add a case for `"in"` that emits `OP_IN`.

### VM changes

Add `OP_IN` handler:
1. Pop haystack (right operand), pop needle (left operand).
2. Switch on haystack type:
   - `VAL_MAP`: call `obj_map_has(&haystack.map, &needle)`. Push `make_bool(result)`.
   - `VAL_ARRAY`: linear scan through `haystack.array->data[0..count-1]`, calling `value_equal(&needle, &elem)` for each. Push `make_bool(found)`.
   - `VAL_STRING`: validate needle is `VAL_STRING`. Use `strstr(haystack.string, needle.string)`. Push `make_bool(result != NULL)`.
   - Other: runtime error "cannot use 'in' with \<type\>".
3. Free needle and haystack.

### Step 1: Write all tests

Write parser tests, compiler error tests, and VM integration tests before any implementation.

**Parser tests** (`tests/test_parser.c`):
- `1 in [1, 2]` → `[BINOP in [NUMBER 1] [ARRAY [NUMBER 1] [NUMBER 2]]]`.
- `"a" in m` → `[BINOP in [STRING a] [IDENT m]]`.
- `1 not in [1, 2]` → `[UNARY not [BINOP in [NUMBER 1] [ARRAY [NUMBER 1] [NUMBER 2]]]]`.
- `not 1 in [1, 2]` → `[UNARY not [BINOP in [NUMBER 1] [ARRAY [NUMBER 1] [NUMBER 2]]]]` (same AST as `not in` form).
- `1 in [1] in [2]` → parse error (non-associative, cannot chain).
- `my in = 5` → parse error (`in` is reserved).

**VM tests** (`tests/test_eval.c`):

Map membership:
- `"a" in {a: 1, b: 2}` → true.
- `"c" in {a: 1, b: 2}` → false.
- `"a" in {}` → false.
- `1 in {[1]: "one"}` → true (number key).
- `true in {[true]: "yes"}` → true (boolean key).
- `"a" in {a: nothing}` → true (key exists even though value is nothing).

Array membership:
- `42 in [1, 2, 42]` → true.
- `99 in [1, 2, 3]` → false.
- `"b" in ["a", "b", "c"]` → true.
- `true in [1, true, "hi"]` → true.
- `1 in []` → false.

String membership:
- `"lo" in "hello"` → true.
- `"xyz" in "hello"` → false.
- `"" in "hello"` → true.
- `"hello" in "hello"` → true.
- `42 in "hello"` → error "in requires a string left operand for string search".

`not in`:
- `10 not in [1, 2, 3]` → true.
- `1 not in [1, 2, 3]` → false.
- `"z" not in {a: 1}` → true.
- `"a" not in {a: 1}` → false.
- `"xyz" not in "hello"` → true.

Error cases:
- `1 in 42` → error "cannot use 'in' with number".
- `1 in true` → error "cannot use 'in' with boolean".
- `1 in nothing` → error "cannot use 'in' with nothing".

Precedence:
- `1 + 1 in [2, 3]` → true (arithmetic binds tighter: `(1+1) in [2,3]`).
- `true and 1 in [1, 2]` → true (`and` binds looser: `true and (1 in [1,2])`).
- `not 5 in [1, 2, 3]` → true (`not` binds looser: `not (5 in [1,2,3])`).

Run `make test && make check` to confirm all new tests fail.

**Files touched**: `tests/test_parser.c`, `tests/test_eval.c`.

### Step 2: Parser — `in` as keyword infix + `not in` sugar

**Parser** (`src/parser.c`):
- Add `"in"` to `is_reserved_keyword()`.
- Add `"in"` to `is_keyword_infix()`: `return token_is_keyword(t, "and") || token_is_keyword(t, "or") || token_is_keyword(t, "in");`
- Add `"in"` to `infix_precedence()`: when `len == 2` and `op` is `"in"`, return 4.
- Add `"in"` to `is_comparison_op()`: when `len == 2` and `op` is `"in"`, return true. This enables the existing non-associativity check in `parse_expr()`.
- Also add `"in"` to the keyword-exclusion lists that `break`, `continue`, and `return` use when checking whether the next token starts a value expression (alongside `"and"`, `"or"`, `"end"`, etc.). This prevents `break in` from being parsed as `break` with value expression `in ...`.
- `not in` handling in `parse_expr()`: in the while loop condition, add a check for the `not in` compound. When `token_is_keyword(&p->current, "not")` is true, peek at the next token. If the next token is `token_is_keyword(&next, "in")`:
  1. Save the line/col of `not`.
  2. Advance past `not`.
  3. Advance past `in`.
  4. Parse right operand with `parse_expr(p, 5)` (precedence 4 + 1, left-associative).
  5. Build `AST_BINOP "in"` with `left` and the right operand.
  6. Wrap in `AST_UNARY "not"`.
  7. Assign to `left` and continue the loop.
  - If `not` is followed by something other than `in`, do NOT enter this path — break out of the loop and let `not` be handled as part of the next expression (it's not a valid infix operator on its own, so the loop naturally stops).

Run `make test && make check`. Parser tests should now pass; VM tests should still fail.

**Files touched**: `src/parser.c`, tests.

**Compiler** (`src/compiler.c`):
- In the function that compiles `AST_BINOP` nodes (likely `compile_binop`): add a check for the operator string `"in"`. Compile left, compile right, emit `OP_IN`.

**VM** (`src/vm.c`):
- Add `OP_IN` handler:
  1. Pop haystack (right), pop needle (left).
  2. Switch on `haystack.type`:
     - `VAL_MAP`: `bool found = obj_map_has(haystack.map, &needle);`. Push `make_bool(found)`.
     - `VAL_ARRAY`: linear scan through `haystack.array->data[0..count-1]`. For each element, call `value_equal(&needle, &elem)`. If any returns true, `found = true` and break. Push `make_bool(found)`.
     - `VAL_STRING`: validate `needle.type == VAL_STRING`. If not → runtime error "in requires a string left operand for string search". Otherwise `bool found = strstr(haystack.string, needle.string) != NULL;`. Push `make_bool(found)`.
     - Other: runtime error "cannot use 'in' with \<type\>" (using `value_type_name(haystack.type)`).
  3. Free needle and haystack.

Run `make test && make check`. All tests should now pass.
