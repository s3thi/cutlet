# `@:` — Zip Arrays into a Map

## Objective

Add the `@:` operator, which zips two parallel arrays into a map. The left array provides keys, the right array provides values. This is the idiomatic way to construct a map from arrays in Cutlet, composing naturally with the `@` meta-operator on arrays.

When done:

- `["a", "b", "c"] @: [1, 2, 3]` → `{a: 1, b: 2, c: 3}`.
- Composes with `@` on arrays: `names @: (scores @* 1.1)`.
- Map inversion: `values(m) @: keys(m)`.
- Mismatched array lengths are a runtime error.
- Non-array operands are a runtime error.
- Invalid key types in the left array are a runtime error.
- Duplicate keys: last occurrence wins.
- `make test && make check` pass.

## Acceptance criteria

- [ ] Compiler recognizes `@:` in infix position (TOK_META with value `":"`), emits `OP_ZIP_MAP`.
- [ ] Compiler rejects `@:` in prefix position with a compile error.

## Constraints and non-goals

- **No `zip()` builtin function.** `@:` is the idiomatic syntax. A function alias can be added later if desired.
- **No unzip.** Use `keys()` and `values()` (from the maps plan) to decompose a map back into arrays.
- **No multi-array zip.** Only two-operand zip (keys, values). Multi-way zip can be built from nested calls.

## Design

### `@:` syntax

`@:` follows the same tokenization path as all `@op` meta-operators. The tokenizer sees `@` followed by `:` (a solo symbol) and emits `TOK_META` with value `":"`.

```cutlet
["alice", "bob", "carol"] @: [92, 87, 95]
# => {alice: 92, bob: 87, carol: 95}

# Compose with other @ operations:
my names = ["alice", "bob"]
my scores = [85, 92]
names @: (scores @* 1.1)
# => {alice: 93.5, bob: 101.2}

# Invert a map:
my m = {a: 1, b: 2, c: 3}
values(m) @: keys(m)
# => {1: a, 2: b, 3: c}
```

### Precedence

`@:` in infix position uses the same precedence logic as other `@op` operators. Since `:` has no established operator precedence, use precedence 6 (same as `+`/`-` and the default for custom `@ident` operators in the arrays plan). This means:

```cutlet
names @: scores @* 1.1
# Parses as: names @: (scores @* 1.1)
# Because @* has higher precedence (7, same as *) than @: (6).
```

### Error cases

- Mismatched lengths: `["a"] @: [1, 2]` → runtime error "array length mismatch in @: (left=1, right=2)".
- Non-array operand: `"a" @: [1]` → runtime error "@: requires two arrays".
- Invalid key type: `[fn(x) is x end] @: [1]` → runtime error "invalid map key type: function".
- Prefix `@:`: `@: [1, 2, 3]` → compile error "':' cannot be used as a reduction operator".

### Duplicate keys

When the left (keys) array contains duplicates, the last occurrence wins — consistent with map literal behavior `{a: 1, a: 2}` → `{a: 2}`.

```cutlet
["a", "b", "a"] @: [1, 2, 3]
# => {a: 3, b: 2}
```

## Architecture changes

### New opcode

| Opcode | Operand | Stack effect | Description |
|--------|---------|-------------|-------------|
| `OP_ZIP_MAP` | — | pop 2 (keys array, values array), push 1 map | Zip two arrays into a map |

### Compiler change

In the compiler's `@` infix handler (the function that compiles `AST_VECTORIZE` nodes — likely named `compile_vectorize` or similar): when the inner operator value is `":"`, instead of emitting `OP_VECTORIZE [op_byte]`, compile left expression, compile right expression, and emit `OP_ZIP_MAP` (no operand byte).

In the compiler's `@` prefix handler (the function that compiles `AST_REDUCE` nodes — likely named `compile_reduce` or similar): when the inner operator value is `":"`, emit a compile error `"':' cannot be used as a reduction operator"`.

Each step follows the required process: tests first, confirm failures, get user confirmation, implement, `make test && make check`.

### Step 1: Add `OP_ZIP_MAP` opcode + disassembler

**Opcode** (`src/chunk.h`):
- Add `OP_ZIP_MAP` to the `OpCode` enum (no operand).

**Disassembler** (`src/chunk.c`):
- Add `OP_ZIP_MAP` to `opcode_name()` → `"OP_ZIP_MAP"`.
- Add a case in the disassembly function to print `OP_ZIP_MAP` as a simple (no-operand) instruction.

**Files touched**: `src/chunk.h`, `src/chunk.c`.

### Step 2: Compiler — recognize `@:` and emit `OP_ZIP_MAP`

**Compiler** (`src/compiler.c`):
- In the function that compiles `AST_VECTORIZE` nodes: add a check at the top. If `node->value` is `":"`, compile `node->left` (push keys array), compile `node->right` (push values array), emit `OP_ZIP_MAP`, and return early (skip the normal `OP_VECTORIZE` path).
- In the function that compiles `AST_REDUCE` nodes: add a check at the top. If `node->value` is `":"`, call `compiler_error(c, "':' cannot be used as a reduction operator")` and return.

**Tests** (`tests/test_eval.c` or `tests/test_parser.c`):
- Parse `["a"] @: [1]` → `[VECTORIZE : [ARRAY [STRING a]] [ARRAY [NUMBER 1]]]` (parser produces AST_VECTORIZE with value ":").
- `@: [1, 2, 3]` → compile error containing "cannot be used as a reduction".

**Files touched**: `src/compiler.c`, tests.

### Step 3: VM handler for `OP_ZIP_MAP`

**VM** (`src/vm.c`):
- Add a case for `OP_ZIP_MAP`:
  1. Pop right operand (values), pop left operand (keys).
  2. Validate both are `VAL_ARRAY`. If not → runtime error "@: requires two arrays".
  3. Validate same length. If not → runtime error "array length mismatch in @: (left=N, right=M)".
  4. Create a new `ObjMap` via `obj_map_new()`.
  5. Iterate from index 0 to length - 1:
     - Validate the key at `keys[i]` is a valid key type (string, number, bool, nothing). If not → runtime error "invalid map key type: \<type\>".
     - Call `obj_map_set()` with `keys[i]` and `values[i]`.
  6. Free the two popped array Values.
  7. Push `make_map(map)`.

**Tests** (`tests/test_eval.c`):
- `["a", "b", "c"] @: [1, 2, 3]` → `{a: 1, b: 2, c: 3}`.
- `[] @: []` → `{}`.
- `["x"] @: [42]` → `{x: 42}`.
- `[1, 2] @: ["one", "two"]` → `{1: one, 2: two}` (number keys).
- `[true, false] @: ["yes", "no"]` → `{true: yes, false: no}` (boolean keys).
- Duplicate keys: `["a", "b", "a"] @: [1, 2, 3]` → `{a: 3, b: 2}`.
- Mismatched lengths: `["a"] @: [1, 2]` → error containing "array length mismatch".
- Non-array operand: `"a" @: [1]` → error containing "requires two arrays".
- Non-array operand: `[1] @: "a"` → error containing "requires two arrays".
- Invalid key type: `[fn(x) is x end] @: [1]` → error containing "invalid map key type".

**Files touched**: `src/vm.c`, tests.

### Step 4: Composability + integration tests

**Composability tests** (`tests/test_eval.c`):
- `my names = ["a", "b"]\nmy scores = [10, 20]\nnames @: scores` → `{a: 10, b: 20}`.
- Compose with `@*`: `["a", "b"] @: ([10, 20] @* 2)` → `{a: 20, b: 40}`.
- Map inversion: `my m = {x: 1, y: 2}\nvalues(m) @: keys(m)` → `{1: x, 2: y}`.
- Round-trip: `my m = {a: 1, b: 2}\nkeys(m) @: values(m)` → `{a: 1, b: 2}`.

**End-to-end** (`tests/test_cli.sh`):
